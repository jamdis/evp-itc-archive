The kind of compression that occurs during an= email attachment is "lossless": it compress the file cleverly without losi= ng information.  For example, if I had a program in some .exe files, a= nd I sent it over email, it could compress it.  But if it lost even on= e bit, the resulting exe file would not execute again after it was uncompre= ssed on the other side.

This kind of compression does things like lo= oking for common patterns in the file, and replacing them with a token.&nbs= p; A rough example would be if it found "Howdy Neighbor" in a file 20 times= , it could replace all the occurances with say $HN1.  And also write a= little dictionary in there someplace that says "$HN1=3DHowdy Neighbor".
When the file gets to its destination it can be uncompressed, and all = occurences of $HN1 are replaced with "Howdy Neighbor".  So it was com= pressed (instead of 14 characters each, it only needed 4, plus the dictiona= ry), but lost no data.

Some formats, like jpeg and mp3. are not loss= less.  Depending on the level of compression, some level of data is lo= st, and so cannot be recovered or reconstructed later.
